# 【Dify新機能】ナレッジパイプライン登場

**🔗 URL**: [【Dify新機能】ナレッジパイプライン登場](https://qiita.com/DifyJapan/items/1f4a076254df203de6c5)  
**📅 Date**: 2025-09-24

## 📄 Original Content

Difyが正式リリースした「ナレッジパイプライン（Knowledge Pipeline）」は、企業の複雑な非構造化データをLLM向けの高品質なコンテキストに変換するオーケストレーション可能なRAGデータ処理パイプライン。従来のRAGが直面していた「データソースの分断」「異種データの解析困難」「処理プロセスのブラックボックス化」の3つの課題を解決し、可視化・標準化されたETLプロセスを提供。キャンバスベースの直感的操作でデータソース接続からベクトル化まで各ステップを独立ノードで表現し、豊富なプラグインエコシステムによる柔軟なカスタマイズを実現。リアルタイムデバッグ、変数インスペクター、テスト実行機能により処理プロセス全体を可観測化し、7種類のプリセットテンプレートで迅速な開発を支援する。

## 🎯 Summary

- **コンテキストエンジニアリング基盤の確立**: 企業データの大部分を占める非構造化ファイル（PDF、PPT、Excel等）をLLM最適化されたコンテキストに変換する体系的エンジニアリングプロセスを実現
- **3層構造のETLパイプライン**: Extract（30種類以上データソース対応）→Transform（解析・強化・チャンキング・ベクトル化）→Load（インデックス構築）の完全可視化された処理フロー
- **プラグインアーキテクチャによる拡張性**: Marketplace提供の豊富なプラグインをレゴブロック的に組み合わせ、業界特化型データ処理パイプラインを自由構築可能
- **完全可観測化されたデバッグ**: テスト実行・変数インスペクター・リアルタイム監視により、従来のブラックボックス化を解消し根本原因の迅速特定を実現
- **エンタープライズ対応と開発効率化**: 7つのプリセットテンプレート、Human-In-The-Loop対応、再利用可能アセット化でビジネスチームと技術チームの協働を促進

## 📊 従来RAGシステムとの比較

| 項目 | 従来RAG | Dify ナレッジパイプライン |
|---|---|---|
| データソース統合 | 個別対応・カスタムコード必要 | プラグインワンクリック導入 |
| 処理プロセス | ブラックボックス化 | 完全可視化・ノード分離 |
| デバッグ方法 | 盲目的・原因特定困難 | リアルタイム変数監視・テスト実行 |
| カスタマイズ性 | 固定的・制限あり | レゴブロック式・完全柔軟 |
| マルチモーダル対応 | 限定的 | テキスト・画像・表・音声統合 |
| チーム協働 | 技術チーム依存 | ビジネスチーム参加可能 |

## 📊 対応データソース・プラグイン一覧

| カテゴリ | 対応サービス・形式 |

## 📊 プリセットテンプレート

| テンプレート名 | 用途 | 特徴 | 適用場面 |
|---|---|---|---|
| General Mode (ECO) | 一般文書処理 | 汎用段落分割・経済的インデックス | 大量文書高速処理 |
| Parent-Child (HQ) | 長文書処理 | 親子階層チャンキング・完全コンテキスト保持 | 技術文書・研究レポート |
| Simple Q&A | 表データQ&A抽出 | 指定列から構造化Q&Aペア生成 | 自然言語データ検索 |
| Complex PDF | 複雑PDF解析 | 画像・表専門抽出 | マルチモーダル検索 |
| Contextual Enrichment | マルチモーダル強化 | LLM活用画像・表テキスト化 | 検索効率向上 |
| Convert to Markdown | 文書形式変換 | Office→Markdown変換 | 処理効率・互換性向上 |
| LLM Generated Q&A | インテリジェントQ&A | 重要情報自動抽出・Q&A化 | 長文正確知識ポイント変換 |

## 📊 Transform層の4ステップ詳細

| ステップ | 機能 | 技術要素 | 品質保証観点 |
|---|---|---|---|
| **Parse（解析）** | ファイルタイプ別最適解析 | OCR・表復元・テキストボックス順序整理 | 複数パーサー並列実行による情報損失防止 |
| **Enrich（強化）** | LLMによるコンテンツ品質向上 | エンティティ抽出・要約・タグ分類・機密マスキング | コンテンツ品質の定量的向上 |
| **Chunking（チャンキング）** | 3戦略対応分割処理 | General・Parent-Child・Q&A戦略 | 文書論理構造保持・最適粒度制御 |
| **Embedding（ベクトル化）** | 要件別モデル選択 | コスト・言語・次元数対応 | プロバイダー柔軟切り替え |

## 💭 My Notes

- 品質保証エンジニアの観点から、「可観測性」と「テスト実行」機能は従来のデータ処理パイプラインに欠けていた重要な品質管理要素
- ETLプロセスの各ステップが独立ノード化されているため、障害分離とボトルネック特定が効率的に行える設計
- 複数パーサー並列実行による情報損失防止は、半導体テストにおける複数測定手法併用によるデータ精度向上と同様のアプローチ
- メカトロニクス分野での技術文書管理・品質マニュアル整備にParent-Child戦略が有効活用できそう
- データ分析の観点から、構造化Q&Aペア生成機能は過去分析結果の知識ベース化に応用可能
