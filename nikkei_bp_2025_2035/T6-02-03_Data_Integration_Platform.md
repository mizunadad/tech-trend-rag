# T6-02-03 データ連携基盤・オープンデータ公開

## Summary（5つの要点）

1. **スマートシティリファレンスアーキテクチャ準拠**: 内閣府が推奨する都市OSの3要件「相互運用性(つながる)」「データ流通(ながれる)」「拡張容易性(つづけられる)」を満たす基盤。異なる部門・組織・都市間でデータ共有・活用が可能
2. **FIWARE(ファイウェア)の普及**: 欧州発のオープンソースデータ連携基盤。NGSI(Next Generation Service Interface)標準APIにより、異なるシステム間でリアルタイムデータを疎結合で連携。日本ではデジタル庁が推奨モジュールとして採用
3. **非パーソナルデータとパーソナルデータの二層構造**: FIWARE Orionが非パーソナルデータ(センサー、交通、環境等)を管理。パーソナルデータは本人同意に基づくPDS(Personal Data Store)で分離管理。個人情報保護とデータ活用を両立
4. **SaaS型・共同利用型の普及**: 浜松市等がSaaS型データ連携基盤を導入。自治体はデータ連携基盤を所有せず利用に徹することで、初期投資・運用負担を削減。加古川市のFIWARE基盤を複数自治体で共同利用する事例も
5. **日本の課題とエストニアX-Roadの対比**: 日本はFIWAREを中心にデータ連携基盤を構築中だが、自治体間・都市間の相互運用性は発展途上。エストニアX-Roadは2001年から全国統一基盤で450組織・3,000サービスを接続し、99%のオンライン利用率を実現

## 具体的プロダクト事例

### 日本企業の先進事例

#### NEC データ利活用基盤サービス(FIWARE準拠) - 高松市
**開発**: 日本電気株式会社(NEC)
- **概要**: FIWAREをベースに品質検証・セキュリティ強化したクラウドサービス。高松市が防災・観光分野で導入
- **主要機能**:
  - 防災データ可視化: 水位センサー・潮位センサーからリアルタイムデータ収集。香川県「かがわ防災Webポータル」のオープンデータと統合し、「高松市ダッシュボード」で可視化
  - IoTセンサー連携: 街中の分散センサーデータをクラウド上で統合管理
  - データ共有・分析・加工: 防災、観光、交通、エネルギー、環境等の多分野データを横断的に活用
- **技術的特徴**:
  - FIWARE Orion Context Brokerで分散データ管理・中継
  - NGSI標準APIで異システム間連携
  - OpenStack、Hadoopベースで低コスト構築
- **実績**: 2011年からFIWARE開発に参画。高松市での実証により防災・観光DX推進
- **リンク**: https://jpn.nec.com/techrep/journal/g18/n01/180106.html

#### 富士通 スマートシティデータ連携基盤
**開発**: 富士通Japan株式会社
- **概要**: FIWARE準拠の都市OSプラットフォーム。非パーソナル+パーソナルデータの両対応
- **主要機能**:
  - FIWARE OrionによるNGSI標準インタフェース
  - パーソナルデータ利活用基盤(PDS): 本人同意(オプトイン)に基づくマイナポータル、医療・健康情報等のパーソナルデータ蓄積・利活用
  - PHR(Personal Health Record)の長年実績を基盤に展開
  - 自治体、民間企業、都市間のデータ連携を実現
- **利用シーン**: ヘルスケア、防災、教育など分野横断サービス。住民は専用ポータルから個別ニーズに応じたサービスを受ける
- **認定**: デジタル庁「デジタル実装の優良事例を支えるサービス/システムのカタログ(第2版)」(2023年12月)掲載
- **リンク**: https://www.fujitsu.com/jp/group/fjj/solutions/business-technology/mobility/smart-city/platform/

#### Code for Japan MoCデータ連携基盤
**開発**: 一般社団法人Code for Japan
- **概要**: FIWAREベース・AWS活用のSaaS型データ連携基盤。スマートシティリファレンスアーキテクチャ準拠
- **導入実績**: 7自治体導入(2024年時点)。デジタル庁「デジタル地方創生サービスカタログ(2024年冬版)」掲載
- **特徴**:
  - 実証実験から段階的展開が可能
  - オープンソース・オープンデータのナレッジ提供(東京都コロナサイト等の開発実績)
  - 市民参加型スマートシティの実現を目指す
- **MoCエコシステム**: データやサービスを集め、エコシステムを形成・促進するプログラム。認証済みサービス・センサーの登録制度あり
- **リンク**: https://www.code4japan.org/activity/moc

#### インテック エリアデータ利活用サービス
**開発**: 株式会社インテック
- **概要**: FIWARE標準搭載のデータ連携基盤。富山県魚津市「魚津モデルスマートシティ」で導入
- **魚津市事例**:
  - GPSセンサーで子どもの登下校見守り
  - データサイエンス教育による地域人材育成
  - 人口減少・担い手不足対策
- **特徴**:
  - クラウドサービスによるスモールスタート可能
  - 広域連携に対応し、安価かつ柔軟な導入
- **提供資料**: スマートシティ進め方ガイドブック、事例集(2024年・2025年版)
- **リンク**: https://www.intec.co.jp/column/smartcity-05.html

#### 加古川市 FIWARE基盤共同利用モデル
**開発**: 加古川市(先行構築)、複数自治体が共同利用
- **概要**: 加古川市が構築したFIWAREベースのデータ連携基盤を複数自治体で共同利用
- **特徴**:
  - オープンデータカタログサイト、ダッシュボード機能を短期間・低コストで導入
  - 各種センサーデータ集約、データ加工・分析対応
  - 自治体の幅広いDXニーズに対応可能な柔軟アーキテクチャ
- **リンク**: デジタル庁「デジタル地方創生サービスカタログ」掲載

#### 浜松市 SaaS型データ連携基盤
**開発**: 浜松市(利用者)、外部SIベンダー(SaaS提供)
- **概要**: 自治体初のSaaS型データ連携基盤導入事例(2023年)
- **特徴**:
  - 浜松市はデータ連携基盤を所有せず、SaaSとして利用
  - FIWARE Orion Context BrokerをAWS PaaS上に構築
  - サービス事業者・アプリ開発者に広く開放し、エコシステム形成
- **狙い**: 初期投資・運用負担削減。民間事業者の参入促進
- **リンク**: https://xtech.nikkei.com/atcl/nxt/column/18/00001/07930/

### グローバルスタンダード

#### FIWARE(ファイウェア) - EU発オープンソース都市OS
**開発**: EU次世代インターネット官民連携プログラム(FI-PPP)。NEC等が2011年から開発参画
- **概要**: スマートシティ・IoTプラットフォーム向けオープンソース基盤ソフトウェア。約40種のモジュール群で構成
- **技術的特徴**:
  - **NGSI(Next Generation Service Interface)**: OMA(Open Mobile Alliance)標準化のオープン規格。APIでデータ送受信
  - **FIWARE Orion Context Broker**: コアコンポーネント。分散データ管理・中継、リアルタイムデータ処理
  - **疎結合設計**: データとアプリケーションが疎結合。システム改修コスト削減、新サービス追加容易
  - **データモデル標準化**: 温度(摂氏/華氏)等の単位変換を自動実施。意味の違いを吸収
  - **モジュール自由組合せ**: データ/コンテキスト管理、IoT、セキュリティ、クラウド等のカテゴリから必要モジュールを選択
- **得意分野**: リアルタイムデータ(センサー、交通、気象等)の収集・管理・提供
- **苦手分野**: 静的な地理空間情報(地図データ等)は別基盤(Geolonia等の地理空間データ連携基盤)と組み合わせが推奨
- **グローバル展開**: 欧州中心に多数の都市・企業で導入。日本でもデジタル庁が推奨モジュールに指定
- **日本での推奨モジュール普及状況(2025年)**:
  - Kong Gateway(APIゲートウェイ): 年間6件の更新・問い合わせ
  - FIWARE Orion(データブローカー): 年間2件の更新・問い合わせ
  - 設定方法・ベストプラクティスに関する実践的質問が中心
- **コミュニティ**: OSS FIWAREコミュニティが活発。簡単インストールツール「FIWARE Big-Bang」等も提供
- **リンク**: https://www.fiware.org/

#### Estonia X-Road(相互運用基盤) vs FIWARE(データ連携基盤)
**比較表**:

| 項目 | 日本FIWARE基盤 | エストニアX-Road |
|------|--------------|-----------------|
| 主目的 | スマートシティのデータ連携 | 行政・民間システムの相互運用 |
| データタイプ | リアルタイムデータ中心 | すべてのデータタイプ(リアルタイム・静的・業務データ) |
| 標準規格 | NGSI(OMA標準) | X-Road独自プロトコル+SOAP/REST API |
| アーキテクチャ | データ中央集約(Orion Broker) | 分散型(各組織が自DBを管理) |
| 導入規模 | 自治体単位で個別導入 | 全国統一基盤(450組織、3,000サービス) |
| 相互運用性 | 自治体間連携は発展途上 | 全国・国際(フィンランド等)で相互接続済 |
| 国際展開 | EU中心 | 25カ国以上(ウクライナ、日本、マレーシア等) |
| 利用実績 | 一部自治体で導入開始 | エストニア国民99%がオンライン利用 |
| サイバー攻撃耐性 | 中央集約型のためSPOFリスク | 分散型でSPOF排除、2007年攻撃でも無停止 |

- **X-Roadの優位性**:
  - 全国統一基盤で自治体間・国境を越えたデータ交換が標準実装
  - 分散型設計によりサイバー攻撃耐性が高く、ウクライナ侵攻下でも稼働継続
  - 20年以上の運用実績と99%のオンライン利用率
- **FIWAREの優位性**:
  - リアルタイムデータ処理に特化。スマートシティ・IoT用途に最適
  - オープンソースで低コスト構築可能。モジュール選択の柔軟性
  - EU中心にグローバルコミュニティが活発
- **日本の課題**: FIWAREを自治体単位で導入中だが、全国統一基盤としての相互運用性は未実装。エストニアX-Roadのような分散型・全国統一・国際相互運用を持つ基盤が不在

## 全体要約の特徴

### 記事内容を的確に把握する5つの要点

1. **スマートシティリファレンスアーキテクチャの3要件**: 内閣府が定義する都市OSの要件。①相互運用性(異なるシステム間でデータ共有)、②データ流通(分野を超えた論理的データ統合)、③拡張容易性(機能追加・更新が容易)。FIWAREはこれらを満たす標準基盤
2. **FIWARE Orion Context Brokerの役割**: 分散配置されたIoTセンサー、オープンデータ、業務システムのデータを一元的に管理・中継。NGSI標準APIでリアルタイムデータを提供。摂氏/華氏等の単位変換も自動実施
3. **非パーソナルデータとパーソナルデータの分離管理**: FIWARE Orionは非パーソナルデータ(交通、気象、環境等)を扱う。パーソナルデータ(マイナポータル、医療・健康情報)はPDS(Personal Data Store)で本人同意に基づき分離管理。個人情報保護法・GDPR対応
4. **SaaS型・共同利用型の普及**: 浜松市がSaaS型で初期投資ゼロ。加古川市のFIWARE基盤を複数自治体で共同利用しコスト削減。自治体はデータ連携基盤を「所有」から「利用」へシフト
5. **データ連携基盤の課題**: FIWAREは導入自治体が増加中だが、①自治体間の相互運用性が不十分(個別導入が主)、②静的データ(地図等)の扱いが苦手(別基盤と組み合わせ必要)、③運用・保守の専門人材不足、④データの質・標準化の未整備

### 日本の立ち位置・弱み・強みを4点で纏めたSummary

#### 強み
1. **FIWARE推奨モジュール採用**: デジタル庁がFIWARE OrionやKong Gatewayを推奨モジュールに指定。自治体が標準に準拠した基盤を導入しやすい環境整備。Data Society Alliance(DSA)がテストベッド提供・ナレッジ共有で支援
2. **SaaS型・共同利用型の先進事例**: 浜松市(SaaS型)、加古川市(共同利用型)等の成功事例が他自治体の参考に。初期投資・運用負担削減モデルが普及の鍵

#### 弱み
1. **自治体間相互運用性の欠如**: FIWAREを導入する自治体は個別最適で進行。全国統一基盤としての相互運用性は未実装。自治体Aのデータを自治体Bが直接利用する仕組みが不在。エストニアX-Roadは全国統一基盤で450組織・3,000サービスを接続し、99%のオンライン利用率を実現。日本は自治体単位の点の取り組みに留まり、線・面への展開が課題
2. **データ標準化の遅れ**: 自治体ごとにデータフォーマット、属性、粒度が異なり、横断的利活用が困難。FIWARE Orionがデータ変換機能を持つが、元データの標準化が不十分では効果限定的。エストニアX-Roadはデータ標準化を前提に全国統一基盤を構築
3. **運用・保守の専門人材不足**: FIWAREはOSSで技術情報が豊富だが、自治体職員のデータエンジニアリングスキル不足。運用監視、セキュリティパッチ適用、障害対応を外部SIベンダーに依存。持続的運用体制の構築が課題
4. **中央集約型アーキテクチャのリスク**: FIWARE Orion Brokerがデータを中央集約するため、単一障害点(SPOF)リスクあり。エストニアX-Roadの分散型設計(各組織が自DBを管理)と比較すると、サイバー攻撃耐性・災害時レジリエンスで劣る

#### 参考:エストニアX-Roadとの比較
- **設計思想の違い**: 日本FIWAREは「データ中央集約・リアルタイム連携」、エストニアX-Roadは「分散型・疎結合相互運用」
- **導入規模の違い**: 日本は自治体単位で個別導入(数十自治体)、エストニアは全国統一基盤(450組織、3,000サービス)
- **国際相互運用性**: X-Roadはエストニアとフィンランドが2018年に相互接続。日本FIWAREは国内完結型
- **サイバー攻撃耐性**: X-Roadは2007年大規模攻撃でもサービス無停止。FIWARE中央集約型はSPOFリスク
