# T6-02-02 自治体共通システム(ガバメントクラウド)

## Summary（5つの要点）

1. **2025年度末までの全国統一移行**: 全国約1,700自治体が2025年度末までに、住民基本台帳・税務・福祉など20の基幹業務について標準準拠システムへ移行を義務付け。デジタル庁が整備するガバメントクラウド上にシステムを集約し、自治体間の相互運用性を向上
2. **マルチクラウド基盤の整備**: AWS、Google Cloud、Microsoft Azure、Oracle Cloud、さくらインターネットの5社がガバメントクラウド事業者として認定。自治体は最適なクラウドを選択可能。共同利用方式によりスケールメリットを追求
3. **当初目標との乖離:コスト増の実態**: 政府は2018年度比で運用コスト3割削減を目標としたが、先行事業では8地域中5地域でランニングコスト増加。標準化に伴う機能追加、セキュリティ対策強化、クラウド移行コストが要因
4. **SE不足と移行困難**: 1,700自治体が一斉移行するため、ガバメントクラウド対応可能なSE不足が深刻化。2024年時点で期限内移行可能な自治体は3割のみとの報道。メインフレーム利用自治体やベンダー撤退ケースでは移行困難
5. **日本の課題とエストニアX-Roadの対比**: 日本は中央集権型クラウド基盤で米国クラウドに依存。エストニアX-Roadは分散型アーキテクチャで各組織が自律的にデータ管理し、単一障害点(SPOF)を排除。サイバー攻撃に対する強靱性と国産技術による主権確保を実現

## 具体的プロダクト事例

### 日本企業の先進事例

#### ガバメントクラウド（デジタル庁主導）
**開発・運営**: デジタル庁(政策立案・標準仕様策定)、地方公共団体情報システム機構(J-LIS、基金運営・財政支援)
- **概要**: 行政の業務システムを統一クラウド上に集約・共通化・標準化した政府共通サービス利用環境。2025年度末までに全自治体が20業務を移行
- **標準化対象20業務**:
  - 住民基本台帳、選挙人名簿管理、国民年金、戸籍、戸籍の附票、印鑑登録
  - 固定資産税、個人住民税、法人住民税、軽自動車税
  - 就学、健康管理、児童手当、児童扶養手当、子ども・子育て支援
  - 生活保護、障害者福祉、介護保険、国民健康保険、後期高齢者医療
- **認定クラウド事業者(2024年時点)**:
  - AWS(先行事業で最多採用)
  - Google Cloud(価格競争力で攻勢)
  - Microsoft Azure
  - Oracle Cloud Infrastructure
  - さくらインターネット(国産クラウド、2024年追加認定)
- **利用形態**:
  - **単独利用方式**: 自治体が個別にクラウド運用管理事業者と契約。統制権は自治体保持
  - **共同利用方式(政府推奨)**: 複数自治体のシステム運用を事業者が一括提供
    - アカウント分離型(大勢): 自治体ごとにAWSアカウント等を分離
    - ネットワーク分離型: ネットワークレベルで分離
    - アプリケーション分離型: アプリケーションレベルで分離
- **財政支援**: J-LIS基金により移行費用・運用費増加分を地方交付税措置で支援
- **移行状況(2024年時点)**:
  - 先行事業12団体がリフト完了、うち6団体が本稼働
  - スケジュール通り移行可能な自治体は約3割(情報公開請求による独自取材結果)
  - 特定移行支援システム(延期措置)該当見込みの自治体も存在
- **課題**:
  - ランニングコスト増加: 先行8地域中5地域でコスト増(最大15.9%減~増加あり)
  - 機能追加・セキュリティ対策強化により従来より高額な見積もり
  - ガバメントクラウド利用料の大口割引・長期継続割引の交渉余地
- **リンク**: https://www.digital.go.jp/policies/local_governments

#### 日立システムズ ADWORLD(ガバメントクラウド対応支援)
**開発**: 株式会社日立システムズ
- **概要**: 自治体向け基幹業務システム「ADWORLD」のガバメントクラウド移行支援サービス
- **支援内容**:
  - **リフト支援**: 仮想空間、ネットワーク、セキュリティ設定、OS構築までを実施。IaC(Infrastructure as Code)活用で再現性・標準化を実現
  - **運用支援**: 24時間365日の運用監視、インシデント対応、セキュリティパッチ適用
  - **ネットワーク支援**: LGWAN(総合行政ネットワーク)とガバメントクラウド間の閉域網接続。ネットワークアカウント兼運用管理補助者、回線運用補助者として対応
- **実績**: 12団体のリフト完了、6団体本稼働。ガバメントクラウド推進センタ(専任SE配置)を2023年10月設立
- **特徴**: 他社業務アプリケーション利用自治体にも対応可能。AWS前提(2024年8月時点)
- **リンク**: https://www.hitachi-systems.com/

#### NTTドコモビジネス クラウド接続ソリューション
**開発**: 株式会社NTTドコモ
- **概要**: ガバメントクラウド先行事業の接続サービスを一手に担う。月間利用可能率99.999%以上のSLA、最大10Gbpsの回線選択肢
- **特徴**: 拠点~クラウド間をセキュアな閉域で接続。多彩なセキュリティ対策でサイバーリスク対応
- **リンク**: https://www.ntt.com/bizon/government-cloud.html

### グローバルスタンダード

#### Estonia X-Road（分散型相互運用基盤）
**開発・運営**: Nordic Institute for Interoperability Solutions (NIIS)
- **概要**: エストニアの分散型データ交換レイヤー。2001年稼働。各省庁が独自データベースを管理し、X-Roadで相互接続
- **日本のガバメントクラウドとの対比**:

| 項目 | 日本ガバメントクラウド | エストニアX-Road |
|------|----------------------|-----------------|
| アーキテクチャ | 中央集権型クラウド基盤 | 分散型データ交換レイヤー |
| データ管理 | クラウド上に集約 | 各組織が自組織のDBを管理 |
| 単一障害点(SPOF) | クラウド基盤がSPOFリスク | 各組織独立でSPOF排除 |
| クラウド事業者 | AWS、Google、Azure等米国依存 | 国産技術・オープンソース |
| サイバー攻撃耐性 | クラウド基盤への攻撃リスク | 分散型で攻撃影響を局所化 |
| 相互運用性 | 標準仕様書で統一 | X-RoadのAPIで疎結合連携 |
| 導入国数 | 日本のみ | 25カ国以上(ウクライナ、マレーシア、日本等) |

- **技術的特徴**:
  - **分散型設計**: 各省庁・自治体が独自システム・DBを運用。X-Roadは「データ交換の高速道路」として機能
  - **相互運用性**: SOAP/REST API対応。最小限の統合労力で異システム間連携
  - **強靱性**: 2007年大規模サイバー攻撃でもサービス無停止。分散型設計が奏功
  - **透明性**: 市民は自分のデータに誰がアクセスしたかをログで確認可能
- **グローバル展開**: 
  - ウクライナ"Trembita"(2019年実装、2022年ロシア侵攻後も稼働継続)
  - フィンランドとエストニアは2018年にX-Road相互接続し、国境を越えたデータ交換実現
  - 日本、マレーシア、ナミビア、ベナンなど25カ国以上で導入
- **利用実績**: 
  - エストニア国民の99%が行政サービスをオンライン利用
  - 月間約2億9,500万リクエスト処理
- **リンク**: https://e-estonia.com/solutions/interoperability-services/x-road/

#### Ukraine Trembita（ウクライナ版X-Road）
**開発**: Cybernetica(エストニア企業)のUnified Exchange Platformベース
- **概要**: 2019年実装。X-Roadをウクライナ向けカスタマイズ
- **実績**: Diiaアプリ(行政サービスアプリ)のバックエンド。2022年ロシア侵攻後も稼働継続し、遠隔行政サービスを維持
- **分散型の強み**: 首都キーウが攻撃されても、各地方のデータベースは独立稼働し、サービス継続可能
- **リンク**: https://nortal.com/

## 全体要約の特徴

### 記事内容を的確に把握する5つの要点

1. **法制化による強制移行**: 2021年9月施行の「地方公共団体情報システムの標準化に関する法律」により、標準化は義務、ガバメントクラウド移行は努力義務(実質的には大多数が移行)
2. **20業務の標準仕様書**: デジタル庁と制度所管省庁が業務ごとの機能要件を定めた標準仕様書を策定(2022年8月全業務揃う)。共通機能標準仕様書、データ要件・連携要件標準仕様書も整備
3. **財政支援と地方交付税措置**: J-LIS基金により移行費用を支援。ガバメントクラウド利用料増加分は地方交付税措置で手当て予定(ただし実際の増加額との乖離が懸念)
4. **SE不足の深刻化**: 全国一斉移行により、クラウド構築・運用対応できるSEが不足。日立システムズ等がガバメントクラウド推進センタを設立し専任SE配置で対応
5. **コスト削減目標と現実の乖離**: 政府は「2018年度比で3割削減」を目標としたが、先行8地域中5地域でコスト増。標準化に伴う機能追加(法改正対応、セキュリティ対策含む)、クラウド利用料、ネットワーク経費が要因。見積内容の精査と事業者への丁寧な説明要請が進行中

### 日本の立ち位置・弱み・強みを4点で纏めたSummary

#### 強み
1. **全国統一の標準化推進**: 法制化により1,700自治体が一斉に標準準拠システムへ移行。これにより将来的には全国レベルでのデータ連携・分析が可能になる(例:人口動態分析、税収予測、災害時の広域連携)
2. **マルチクラウド戦略**: AWS、Google、Azure、Oracle、さくらインターネットの5社を認定し、自治体が選択可能。ベンダーロックイン回避とコスト競争促進を狙う

#### 弱み
1. **中央集権型設計のリスク**: ガバメントクラウド基盤への一極集中により、単一障害点(SPOF)リスクが高まる。クラウド基盤へのサイバー攻撃や大規模障害発生時に全国の行政サービスが停止する恐れ。エストニアX-Roadの分散型設計とは対照的
2. **米国クラウド依存とデジタル主権の欠如**: AWS、Google、Azureなど米国クラウド事業者への依存が高く、データ主権(Digital Sovereignty)の観点で懸念。国産クラウドはさくらインターネットのみで競争力・シェアは限定的。米国政府によるデータアクセス要求(CLOUD Act)やサービス停止リスクに脆弱
3. **コスト削減目標の未達成**: 「3割削減」目標に対し、先行事業で8地域中5地域がコスト増。標準化に伴う機能追加(現行にない法改正対応、セキュリティ対策)、クラウド利用料の見積もり不透明性、ネットワーク経費増が要因。100年かけてもコスト回収できないとの試算もあり、財政圧迫の懸念
4. **SE不足と移行遅延**: 2024年時点でスケジュール通り移行可能な自治体は3割のみ。メインフレーム利用自治体、ベンダー撤退・廃業により移行困難なケースも。特定移行支援システムとして延期措置を取る自治体が続出し、当初の「2025年度末」目標達成は困難な情勢

#### 参考:エストニアX-Roadとの比較から見る日本の課題
- **設計思想の違い**: 日本は「クラウド統合」、エストニアは「分散型相互運用」。エストニアは各組織が自組織のシステム・DBを管理し、X-Roadはデータ交換のみを担当。これにより単一障害点を排除
- **サイバー攻撃耐性**: エストニアは2007年大規模サイバー攻撃、ウクライナは2022年ロシア侵攻下でもサービス無停止。日本のガバメントクラウドは同レベルの強靱性実証が未済
- **国産技術の有無**: X-Roadはオープンソースで25カ国以上が導入。日本のガバメントクラウドは米国クラウド依存
- **国際相互運用性**: エストニアとフィンランドは2018年にX-Road相互接続。日本のガバメントクラウドは国内完結型で国際連携未実装
