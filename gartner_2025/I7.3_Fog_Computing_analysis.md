# Fog Computing（フォグコンピューティング）

## 📊 ハイプサイクル位置情報

**位置**: 🔧 Innovation Trigger（技術革新の初期段階）  
**実用化時期**: 🔵 2-5年で実用化  
**ハイプサイクル**: Gartner Hype Cycle for Infrastructure Strategy, 2025

---

## 📝 5つの要点サマリー

### 1️⃣ **定義と位置づけ**
Fog Computingは、エッジデバイスとクラウドの間に配置される中間層コンピューティングアーキテクチャ。IoTデバイスで生成されたデータをエッジで前処理し、必要なデータのみをクラウドに送信することで、レイテンシ削減、帯域幅最適化、リアルタイム処理を実現。Cisco Systemsが2012年に提唱した概念で、OpenFog Consortiumを通じて標準化が進行中。

### 2️⃣ **ICテスト・品質保証分野への影響**
半導体テスト環境において、テスト装置からのリアルタイムデータをFog層で前処理することで、異常検知の即応性が向上。大量のテストログをクラウドに送信せず、Fog層でフィルタリング・集約することで、ネットワーク負荷を50-80%削減可能。特にウェハーレベルテストや最終テストにおける不良パターンのリアルタイム解析に有効。品質管理データの分散処理により、工場間でのデータ共有と分析のスピードが向上。

### 3️⃣ **メカトロニクス製品への応用**
ロボティクス、産業機械、自動車などのメカトロニクス製品において、制御システムとクラウドAIの間にFog層を配置することで、リアルタイム制御の応答性を維持しながらクラウドベースの高度な分析を活用可能。例えば、産業用ロボットの動作データをFog層で前処理し、予知保全アルゴリズムを実行することで、故障予測精度が向上。製造現場のMES（Manufacturing Execution System）とクラウドERPの間にFog層を配置し、リアルタイム生産管理を実現。

### 4️⃣ **技術的課題と実装上の障壁**
Fog Computing導入の主な課題は、①Fog層のハードウェア選定とコスト（エッジサーバー、ゲートウェイ機器）、②分散システムのオーケストレーション（Kubernetes、Docker等のコンテナ管理）、③セキュリティ管理（Fog層での認証・暗号化）、④既存システムとの統合。特に製造現場では、レガシーPLC（Programmable Logic Controller）やFAシステムとの接続が技術的ハードルとなる。また、Fog層でのデータ処理ロジック（どこまでFogで処理し、何をクラウドに送るか）の設計が複雑化する傾向。

### 5️⃣ **エッジコンピューティングとの違い**
Edge Computingはデバイス自体またはその最も近い場所での処理を指すのに対し、Fog Computingはエッジとクラウドの間の「階層的な分散アーキテクチャ」を提供。Fogは複数のエッジデバイスからのデータを集約・調整する役割を担い、より広範なネットワーク管理とオーケストレーションが可能。実用上は、Edge ComputingがFog Computingの一部として位置づけられることが多く、用語の境界は曖昧化している。Gartnerの分類では、Fog ComputingはEdge Computingよりも統合的・アーキテクチャ的な概念として扱われている。

## 📊 全体要約の特徴（5つの要点）

### 1. **階層的分散アーキテクチャの確立**
Fog Computingは、単なるエッジコンピューティングではなく、IoTデバイス→Edge→Fog→Cloudという階層的なデータ処理フレームワークを提供。製造業においては、現場レベル（Edge）、工場レベル（Fog）、企業レベル（Cloud）の3層構造が最適解となるケースが多く、各層での処理内容を適切に分散することで、レイテンシ、コスト、セキュリティのバランスを最適化。

### 2. **リアルタイム性とクラウドAIの両立**
ICテストや製造現場では、ミリ秒単位の制御応答が必要な一方、大規模データ分析や複雑な機械学習モデルはクラウドで実行する必要がある。Fog層がこの「リアルタイム性」と「高度なAI処理」のトレードオフを解決する中間層として機能。例えば、テスト装置の制御はEdgeで実行し、テストパターンの最適化判断はFogで行い、長期トレンド分析はCloudで実施するという役割分担が可能。

### 3. **データ主権と帯域幅最適化**
製造データやテストデータをすべてクラウドに送信することは、①ネットワーク帯域の制約、②データ主権・プライバシー規制（GDPR、中国サイバーセキュリティ法等）、③クラウドコストの観点から現実的でない。Fog層で前処理・フィルタリングすることで、クラウド送信データ量を90%以上削減できるケースもあり、特に海外工場からの大量テストデータ転送において効果的。

### 4. **予知保全とリアルタイム品質管理への応用**
半導体製造装置やメカトロニクス製品において、Fog Computingは予知保全の精度向上に貢献。エッジで収集したセンサーデータをFog層の機械学習モデルで解析し、異常予兆を数秒以内に検知。従来のクラウドベース予知保全では検知に数分～数時間かかったが、Fog層での推論により即応性が劇的に向上。品質管理では、リアルタイムSPC（Statistical Process Control）をFog層で実行し、不良発生を未然に防止。

### 5. **標準化の進展とエコシステムの構築**
OpenFog Consortium（現在はIndustrial Internet Consortiumに統合）が策定したOpenFog Reference Architectureが業界標準として浸透しつつある。Cisco、Intel、Microsoft、Dell、日立など主要ベンダーがこの標準に準拠したプロダクトを提供開始。しかし、「Fog」と「Edge」の用語が混同されやすく、マーケティング上の混乱も見られる。Gartnerが2025年版でFog ComputingをInnovation Trigger段階に位置づけた背景には、技術的成熟度と市場での認知度のギャップがある。

## 🇯🇵 日本の立ち位置・強み・弱み（4点サマリー）

### 1️⃣ **製造業IoT基盤での強み - 実装力と品質管理ノウハウ**
日本は製造業におけるIoT導入で先行しており、特にFA（Factory Automation）システム、産業用ロボット、製造実行システム（MES）の統合において世界トップレベル。富士通COLMINA、日立Lumada、NECのIoTプラットフォームなど、Fog Computing概念を実装した製品が既に市場投入されている。オムロン、ファナック、安川電機などの産業機器メーカーも、エッジインテリジェンス機能を自社製品に組み込み始めている。品質管理（QC）、カイゼン文化が根付いており、Fog層でのリアルタイム品質監視・SPC実行は日本企業の得意領域。

### 2️⃣ **弱み: グローバル標準化とエコシステム構築への関与不足**
OpenFog Consortium設立時には日立が参加したものの、Cisco、Intel、Microsoftなどの欧米企業が標準化をリード。日本企業は「実装」では強いが、「標準策定」「エコシステム構築」では存在感が薄い。AWS IoT Greengrass、Azure IoT Edge、Cisco IOxといったグローバルプラットフォームに対し、日本発のオープンスタンダードは不在。結果として、日本企業は欧米プラットフォームの採用者（ユーザー）に留まり、ルールメイカーになれていない。

### 3️⃣ **レガシーシステムとの統合力は強いが、クラウドネイティブ化が遅れ**
日本の製造現場には、20-30年前のPLC、SCADA、FAシステムが多数稼働しており、これらとFog層を接続する技術的ノウハウは豊富。しかし、Kubernetes、Docker、マイクロサービスといったクラウドネイティブ技術の習熟度が欧米に比べて低く、Fog層のオーケストレーション基盤構築で遅れ。特にDevOpsやCI/CDパイプラインを製造業IoTに適用する実践例が少なく、アジャイルな開発サイクルが回っていない。

### 4️⃣ **半導体・エレクトロニクス分野での応用余地は大きい**
半導体製造装置（東京エレクトロン、SCREENホールディングス、アドバンテスト等）、テスター、検査装置において、Fog Computing導入による品質向上・コスト削減余地が大きい。ウェハーレベルテスト、最終テスト、信頼性試験で生成される膨大なデータをFog層で前処理し、AIによる不良予測精度を向上させることが可能。しかし、半導体業界は保守的で新技術導入に慎重なため、実証実験から本格導入までのタイムラグが長い傾向。Rapidus等の新興半導体ファウンドリがFog Computing活用を前提とした「スマートファブ」構想を掲げることで、日本の半導体産業全体のデジタル化が加速する可能性がある。
